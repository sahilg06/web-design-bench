Build your first workflow

Welcome to FlowSync — the fastest way to automate anything from webhook processing to multi-system data pipelines. This guide takes you from zero to a production workflow in under five minutes.

New to automation? Start with the visual studio at app.flowsync.io — everything in this guide can be done point-and-click.

Quickstart

Follow these four steps to ship a workflow that watches an inbox, extracts structured data, and posts a Slack notification.

1

Install the FlowSync CLI

The CLI lets you version, test, and deploy workflows from your local machine or CI pipeline.

# macOS & Linux
curl -fsSL https://get.flowsync.io | sh

# Verify installation
flowsync --version
2

Authenticate with your workspace

Log in with SSO or an API token. Tokens are scoped per environment and rotate automatically.

flowsync auth login
# Opens browser → returns with a scoped token

flowsync workspace use acme-production
3

Scaffold your first workflow

The init command generates a starter workflow with a trigger, action, and test fixtures.

flowsync init inbox-to-slack
# Creates ./inbox-to-slack/workflow.yaml

cd inbox-to-slack
flowsync test
4

Deploy and monitor

Push to production, then watch the trace stream in real time — every payload, every retry, every branch.

flowsync deploy --env production
# ✓ Workflow "inbox-to-slack" is live

flowsync logs --tail
# Live stream of execution traces

Core concept: Workflows

A workflow is a directed graph of nodes: exactly one trigger, one or more actions, and optional control-flow nodes (branch, loop, delay, filter). Every workflow is declared in YAML — or built visually and exported to YAML.

# workflow.yaml
name: inbox-to-slack
on:
  gmail.message.received:
    filter: "from:*@customer.com"

steps:
  - extract:
      model: "claude-3-5-sonnet"
      schema: ./schemas/ticket.json

  - slack.postMessage:
      channel: "#support-triage"
      text: "{{ extract.summary }}"

Triggers

Triggers are the entry points that kick off a workflow. FlowSync supports three trigger families:

  • ✓ Event triggers — Slack messages, Gmail arrivals, Salesforce updates, GitHub pushes
  • ✓ Schedule triggers — cron-style expressions with timezone support
  • ✓ HTTP triggers — signed webhooks with per-workflow endpoints

REST API

Every workflow can be invoked, listed, or paused via our REST API. All endpoints require a bearer token and return JSON.

POST /v1/workflows/inbox-to-slack/run
Authorization: Bearer $FLOWSYNC_TOKEN
Content-Type: application/json

{
  "input": {
    "subject": "Refund request #4821",
    "body": "Order arrived damaged..."
  }
}
Rate limits. Starter plans allow 60 req/min. Pro allows 1,200 req/min. Enterprise is negotiable. All endpoints support the Retry-After header on 429 responses.

Node.js SDK

Prefer code? Our SDK wraps the REST API with typed clients for TypeScript and Python.

import { FlowSync } from "@flowsync/sdk";

const client = new FlowSync({ token: process.env.FLOWSYNC_TOKEN });

const run = await client.workflows.run("inbox-to-slack", {
  input: { subject: "Refund", body: "..." }
});

console.log(run.status);   // "succeeded"
console.log(run.output);   // { ticketId: 4821, priority: "high" }

Error handling

Every failed step returns a structured error with a stable code, message, and retry hint. You can catch errors globally with an on_error block, or per-step with a fallback branch.

steps:
  - salesforce.upsert:
      object: "Contact"
      on_error:
        - slack.postMessage:
            channel: "#ops-alerts"
            text: "CRM sync failed: {{ error.message }}"
